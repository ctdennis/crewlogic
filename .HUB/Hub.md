## Update protocol (check в†’ update в†’ recheck)

1. **Check (before a change):** read the relevant row here to confirm current state before starting work.
2. **Update (during the change):** in the *same commit* as the change, update the row вЂ” Build / Prod /
   Version / Open items вЂ” so the tracker never lags the code.
3. **Recheck (after the change):** verify the change landed as described and stamp **Last verified** with
   today's date. If you couldn't verify a field (e.g. prod data), say so in Notes rather than guessing.

**Freshness rule:** trust an entry whose *Last verified* is recent. If it's old or the code in that area
changed since, re-verify before relying on it. Never overstate prod rollout вЂ” "code deployed" в‰  "in active
use by a prod tenant"; keep those distinct.

**Status legend:** `Done` В· `In progress` В· `Planned` В· `Not started` В· `Dropped`
**Prod rollout:** `Live` (in prod & in use) В· `Deployed` (code/fn in prod, not yet exercised by a prod tenant) В· `Dev only` В· `n/a`
